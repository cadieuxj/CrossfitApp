ApexAI Athletics is a B2C Android CrossFit coaching app targeting Canadian athletes including Quebec residents.

**Why:** Pre-launch security audit requested 2026-03-28. Single initial commit — no production deployment yet.

**How to apply:** All findings should be treated as pre-production remediation priorities. No breach notification urgency, but Law 25 compliance must be in place before any Quebec user onboards.

## Tech Stack
- Android (Kotlin, Jetpack Compose, CameraX, MediaPipe, Health Connect)
- Backend: FastAPI (Python 3.11+), deployed on cloud run/Railway
- Database: Supabase (PostgreSQL 15+) with Row Level Security
- AI: Google Gemini 1.5 Pro (video analysis) + Gemini 2.0 Flash (image generation)
- Auth: Supabase Auth (email/password, HS256 JWT)
- Storage: Supabase Storage (private buckets: "videos", "corrections")
- Dependency injection: Hilt
- Navigation: Jetpack Navigation Compose

## Key Data Flows
1. User registers (email + display name) -> Supabase Auth -> profiles trigger
2. User grants Health Connect permissions -> HRV, sleep, resting HR synced to health_snapshots table
3. User records video of workout -> uploaded to FastAPI -> stored to temp file -> sent to Gemini File API -> analyzed -> coaching_reports + movement_faults written to Supabase -> video deleted from temp/Gemini
4. ACWR readiness score calculated via Supabase RPC (calculate_readiness) using results + health_snapshots
5. Correction images generated by Gemini Flash -> stored in Supabase Storage (corrections bucket, signed URLs, 1-hour TTL)

## Authentication Architecture
- Supabase Auth handles JWT issuance (HS256, audience="authenticated")
- FastAPI validates JWT via python-jose using SUPABASE_JWT_SECRET
- FastAPI uses service_role_key (bypasses RLS) for all server-side writes
- Android uses supabase-kt client library

## Known Security Controls Already Implemented
- RLS enabled on all user-owned tables with auth.uid() ownership checks
- athlete_id/JWT sub claim cross-check on POST /analyze-video
- Movement type allowlist validation (server-side)
- Content-type validation for video uploads
- File size streaming check (500 MB max)
- Rate limiting on /analyze-video: 10/hour via slowapi (IP-based)
- android:allowBackup="false" in manifest
- backup_rules.xml + data_extraction_rules.xml exclude sharedpref, database, files
- FileProvider exported="false" with grantUriPermissions="true"
- Supabase Storage signed URLs for corrections (1-hour TTL)
- Gemini file cleanup after analysis (best-effort)
- Structured error envelope (no stack traces in responses)
- Docs/redoc only exposed when DEBUG=true
- RECORD_AUDIO permission present (video has audio)

## Third-Party Data Sharing (Law 25 Implications)
- Google Gemini API: video of athletes (biometric data) and workout context sent to Google servers outside Canada
- Supabase: all personal data stored (likely US/EU servers — jurisdiction must be verified)
- Sentry: error tracking SDK — may capture PII in error contexts
- Google Android Health Connect: data read from device health store (on-device, not shared to Google by this app)
